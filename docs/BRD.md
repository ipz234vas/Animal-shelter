# BUSINESS REQUIREMENTS DOCUMENT (BRD)

## 1. Вступ (Introduction)
* **Мета документа:** Цей документ описує бізнес-вимоги до розробки офіційного сайту для притулку тварин. Він визначає основні бізнес-цілі, цільову аудиторію та очікувану цінність від впровадження продукту.
* **Бізнес-ціль (Business Goal):** Вирішення проблеми обмеженості притулків у цифровій комунікації. Створення платформи дозволить автоматизувати процес пошуку власників для безпритульних тварин, збільшити кількість успішних адопцій та підвищити рівень довіри до притулку через публікацію історій успіху.

## 2. Опис предметної області (Business Context)
Проблема безпритульних тварин в Україні є соціально значущою. Сучасні притулки потребують незалежних цифрових інструментів для презентації своїх підопічних. Аналіз ринку (на прикладі платформ HappyPaw, Cipiyc, Hatul-Madan) показує, що користувачі очікують зручної фільтрації, детальної інформації про тварин та прозорого процесу подачі заявок. Наявність власної системи керування вмістом (CMS) дозволить притулку бути незалежним від сторонніх платформ та гнучко керувати своїми процесами.

## 3. Цільова аудиторія та Ролі (Stakeholders & User Roles)
* **Потенційні усиновлювачі (Гості / Неавторизовані користувачі):** Шукають тварину за певними критеріями (вид, вік, характер), переглядають історії успіху та читають корисні матеріали в блозі.
* **Авторизовані користувачі (Усиновлювачі):** Мають намір офіційно подати заявку на адопцію, відстежувати її статус та ділитися відгуками (розділ "Ми вдома") після успішного всиновлення.
* **Адміністрація притулку (Адміністратори / Модератори):** Працівники або волонтери, які наповнюють каталог тварин, обробляють заявки на адопцію, модерують відгуки та публікують статті у блог.

## 4. Конкурентні переваги (Competitive Advantages)
На основі аналізу існуючих рішень у системі закладено наступні переваги:
* **Оптимізований UX/UI:** На відміну від деяких аналогів, реалізовано повноцінну сторінкову навігацію (пагінацію) замість кнопок "Показати ще", що покращує користувацький досвід та продуктивність.
* **Комплексність:** Об'єднання каталогу, системи заявок, блогу та історій успіху в єдину екосистему.
* **Повна незалежність:** Відмова від готових комерційних CMS на користь власного оптимізованого ядра гарантує повний контроль над даними та відсутність зайвого функціонального навантаження.

## 5. Межі проєкту (Project Scope)
### 5.1. Що входить у проєкт (In-Scope)
* Публічний каталог тварин із системою розширеної фільтрації.
* Система реєстрації та особистих кабінетів користувачів.
* Модуль подання та обробки заявок на адопцію.
* Розділ "Ми вдома" для публікації успішних історій.
* Інформаційний блог.
* Комплексна адміністративна панель (CMS) для керування всіма сутностями сайту.

### 5.2. Що НЕ входить у проєкт (Out-of-Scope)
* **Фінансові транзакції:** Система не обробляє прямі благодійні внески (донати) через інтегровані платіжні шлюзи.
* **Онлайн-чат:** Комунікація між користувачами та адміністрацією здійснюється виключно через систему статусів заявок та залишені контакти, без вбудованого чату реального часу.
* **Нативний мобільний додаток:** Доступ до сервісу з мобільних пристроїв забезпечується через адаптивну веб-версію (HTML5/Bootstrap 5), окремі додатки для iOS/Android не розробляються.

## 6. Високорівневі функціональні вимоги (High-Level Functional Requirements)
1. Система повинна дозволяти відвідувачам переглядати тварин та застосовувати фільтри за видом, віком, статтю та тегами без необхідності реєстрації.
2. Система повинна забезпечувати можливість авторизованим користувачам подавати заявки на всиновлення конкретної тварини.
3. Система повинна дозволяти користувачам публікувати відгуки про успішну адопцію після модерації.
4. Система повинна містити адміністративну панель для керування каталогом тварин, контентом блогу, користувачами та заявками.
